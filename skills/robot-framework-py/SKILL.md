# Robot Framework (Python) — Browser Library Playbook

Use this skill to produce deterministic, maintainable Robot Framework code with enterprise-ready patterns.

## Apply this project structure by default

```text
common
test/robotframework/config
test/robotframework/resources
test/robotframework/test-data
test/robotframework/testsuites
```

- Place suite files in `test/robotframework/testsuites/`.
- Place reusable keywords for suites in `test/robotframework/resources/`.
- Place custom Python keyword libraries and very common keywords in `common/`.
- Place environment-specific variable configurations in `test/robotframework/config/`.

## Available scripts

- **`scripts/list_project_keywords.py`** — Lists all Robot Framework user keywords in the project

## Universal golden patterns

- Keep test cases short; place repeated behavior in resource keywords.
- Use `Suite Setup`/`Suite Teardown` for lifecycle management.
- Use deterministic assertions (`Should Be Equal*`, `Should Contain`, `Should Not Be Empty`).
- Prefer stable local fixtures/mocks over live external systems.
- Keep secrets in environment variables or secure variable files.
- Use consistent tags: `IC-E2E-UI-<6 bit random numbers>`.
- Check existed keywords by using scripts `scripts/list_project_keywords.py`, always reuse existing keywords in `common/` or `test/robotframework/resources/` to avoid duplication and ensure consistency.
- Remove the recording traces and unnecessary keywords in the generated code, and refactor the code by following the above patterns to make it more maintainable and stable.
- Execution logs output is in `logs/others/output.xml` can be used to fix code issues and improve the code quality by following the above patterns.

## Library guidance

### Browser Library (Playwright)

**Recommended patterns**
- Reuse `Open Browser To Login` keyword from `common/keywords/login.resource` in UI cases.
- Prefer role/text/test-id selectors.
- Use context/page keywords via resources.

**Common pitfalls**
- Forgetting `rfbrowser init` in fresh environments.
- Recreating browser per test when suite-level fixture is enough.
- Using unstable CSS selectors.

**Do / Don't**
- Do: use trace/screenshot on failure hooks where needed.
- Don't: mix Selenium and Browser keywords in one suite.

**Minimal working example**
```robotframework
*** Settings ***
Resource    common/keywords/login.resource
Resource    common/keywords/browser_helper.resource

*** Variables ***
${BROWSER_TIMEOUT}    30s

*** Test Cases ***
Open Browser To Login Page
    [Tags]    IC-E2E-UI
    Open Browser To Login
    Set Browser Timeout    ${BROWSER_TIMEOUT}

```

**Environment Variables Settings**
Once installation is finished successfully, please check whether following variables are set into "Environment Variables" of system.

- IED_USER_ID
- IED_USER_PASSWORD
- EA_EDGE_MANAGEMENT_URL
- EA_EDGE_MANAGEMENT_USERNAME
- EA_EDGE_MANAGEMENT_PASSWORD
- AUTO_TEST_ENV (dev/staging/prod)
- CONF_RELATIVE_LOCATION (optional: if you want to change default configuration details relative location `test/robotframework/config`)

The `AUTO_TEST_ENV` setting is utilized to determine which specific `configuration_details.*.json` file will be loaded
for testing

- 'configuration_details.prod.json' is default/production configuration file when `AUTO_TEST_ENV` is not set or set
  as `prod`
- 'configuration_details.staging.json' is staging configuration file when `AUTO_TEST_ENV` is set as `staging`
- 'configuration_details.staging.json' is development configuration file when `AUTO_TEST_ENV` is set as `dev`

## Microsoft-style testing strategy and RALF loop

- **Understand/Plan**: map prompts to expected patterns and forbidden patterns.
- **Build**: keep reusable resources and variable layering.
- **Validate**: add scenario-driven checks and quality gates in CI.
- **Iterate** with **RALF**:
  1. **Review** failures and inconsistency reports.
  2. **Analyze** root causes (guidance gap, example drift, weak assertions).
  3. **Learn** by codifying conventions and anti-patterns.
  4. **Fix** docs/examples/tests and re-run quality checks.

## Execute with this checklist

1. Identify output type (`.robot`, `.resource`, variable file, or Python library).
2. Choose target library and fixture strategy.
3. Generate deterministic assertions and reusable keywords.
4. Refactor duplication into resources.
5. Add/refresh scenario checks and acceptance rules.
6. Ensure CI checks are green.
